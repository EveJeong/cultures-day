## 기능 2: 사용자·팀 관리

### 2.1 목적

운영자가 행사 전/중에 **참가자 명단과 팀 배정**을 등록·수정한다.
인증 플로우(`01-auth-design.md`)의 전제인 "사전 배정된 `users` 문서"를 만드는 화면이며,
참가자 비번 초기화도 여기서 수행한다.

### 2.2 데이터 모델

```ts
// 팀 (편집 가능한 데이터로 승격)
teams/{teamId} = {
  id: 'J' | 'I' | 'L',
  name: string,    // 표시 이름 (편집 가능, 예: "번개팀")
  color: string,   // 팀 색 (편집 가능, hex)
  // 점수는 저장하지 않음 — scoreLog 유효 항목 합계로 파생 (기능 7.2)
}

// 사용자 (인증 문서와 동일 스키마)
users/{이름} = {
  name: string,
  teamId: 'J' | 'I' | 'L',
  nickname: string | null,
  passwordHash: string | null,
  isAdmin: boolean,
}
```

### 2.3 참가자 동작

| 동작 | 설명 |
| --- | --- |
| **추가(일괄)** | 팀 선택 → 이름들을 줄바꿈으로 붙여넣기 → 한 번에 `users` 문서 생성. 단건 추가도 같은 폼. 생성 시 `passwordHash:null`, `nickname:null`, `isAdmin:false` |
| **팀 변경** | 해당 유저의 `teamId` 수정 |
| **이름 변경** | 이름이 docId이므로 *새 문서 재생성*: 새 이름으로 문서 생성 → 기존 필드(팀·비번·닉네임) 이관 → 옛 문서 삭제 |
| **비번 초기화** | `passwordHash → null`. 해당자는 다음 로그인 시 재설정(B) 분기로 진입 |
| **삭제** | `users/{이름}` 문서 제거 |

### 2.4 이름 변경 규칙 (재생성)

이름은 고유 docId라서, 변경은 "삭제 후 재등록"이 아니라 **데이터 이관을 동반한 재생성**으로 처리한다.

```
1. 대상 새 이름이 이미 존재하면 → 거부 ("이미 사용 중인 이름")
2. 새 docId(새 이름)로 문서 생성, 기존 { teamId, passwordHash, nickname } 이관
3. 옛 docId 삭제
```

- 가입 완료자도 비번·닉네임이 보존되어 재로그인 시 그대로 사용 가능.
- ⚠️ 해당 사용자가 **로그인 중이면** 세션키(=이름)가 바뀌므로 **재로그인 필요**. 가급적 본인 로그인 전에 정정하도록 안내한다.

### 2.5 팀 관리 (편집 가능)

- 팀 **개수는 3개 고정**(J·I·L). 추가/삭제는 없음.
- 각 팀의 **표시 이름(`name`)·색(`color`)** 을 운영자가 UI에서 편집.
- 편집 결과는 진행 페이지의 리더보드·점수판에 즉시 반영(`onSnapshot`).

### 2.6 현황 표시

- **팀별 그룹핑** 명단. 각 행에 이름·닉네임·가입상태(가입완료/미가입) 표시.
- 팀별 **인원수** 카운트.
- `admin` 계정(`isAdmin:true`)은 명단에서 **숨김/보호** — 실수로 삭제·수정 방지.

### 2.7 검증 규칙

| 항목 | 규칙 |
| --- | --- |
| 일괄 입력 | 빈 줄 무시, 앞뒤 공백 트림, **중복 이름은 건너뛰고 결과 요약**("N명 추가, M명 중복 제외") |
| 이름 | 빈 값 불가, 기존과 중복 불가 |
| 팀 색 | 유효한 hex 값 |

### 2.8 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 일괄 등록 중 일부 이름이 이미 존재 | 존재하는 이름은 건너뛰고, 추가/제외 건수를 요약 표시 |
| 이름 변경 대상이 이미 존재 | 거부 + 안내 |
| 가입 완료자 삭제 | 확인 다이얼로그(되돌릴 수 없음 경고) |
| 팀에 인원 0명 | 허용 (경고만) |

## 기능 3: 게임 진행 제어

### 3.1 목적

9개 게임을 순서대로 진행하면서 "지금 어느 게임의 어느 단계"인지를 운영자가 제어하고,
그 상태가 모든 진행 페이지(`03-display-design.md`)에 실시간 반영되게 한다.
진행 제어는 **상위 9종 게임 + 단계**까지만 다룬다. (퀴즈 5종 문제 전환은 기능 4, 보드게임·릴레이게임
세부 종목 점수는 기능 5)

### 3.2 상태 모델

```ts
state/current = {
  currentGameId: string,                   // 현재 진행 중인 게임
  phase: 'intro' | 'playing' | 'result',   // 현재 단계
}
```

단일 문서이며, 운영자가 변경하면 진행 페이지가 `onSnapshot`으로 즉시 따라온다.

### 3.3 단계(phase) 정의

| phase | 의미 | 진행 페이지 표시 |
| --- | --- | --- |
| `intro` | 게임 소개 | 게임 이름·규칙·**점수 산정 방식**을 크게 (룰 숙지용) |
| `playing` | 진행 중 | 퀴즈면 현재 문제 + 팀 점수판, 그 외엔 점수판 중심 |
| `result` | 결과 발표 | 이 게임에서 각 팀이 얻은 점수·순위 |

### 3.4 제공 동작

| 동작 | 설명 |
| --- | --- |
| **다음/이전 게임** | 정의된 순서대로 이동 |
| **자유 점프** | 게임 목록에서 아무 게임이나 직접 선택해 이동 |
| **단계 전환** | `인트로 → 진행 → 결과` 버튼. 자유롭게 왕복 가능 |
| **게임 목록 표시** | 정의 순서대로 표시, 현재 게임 하이라이트 |

- **게임을 바꾸면 진행 상태를 리셋**한다(새 게임은 항상 소개부터):
  - `phase = 'intro'`
  - 퀴즈 필드: `currentQuestionId = null`, `quizScreen = 'q1'`
  - 제시어 필드: `promptScreen = null`, `promptAssignment = null`, `promptTeamOrder = null`,
    `currentTeamId = null`, `currentPromptId = null`
  - 타이머: 새 게임의 `games.timer` 기본값으로 `idle` 리셋(§3.6)
  - → stale 배정/문제 포인터가 다른 게임으로 새어 들어가는 것을 방지(자유 점프 안전).
- *(선택)* 점수 기록(`scoreLog`)이 있는 게임엔 목록에 "완료" 뱃지 표시.

### 3.5 연결점

- `result` 단계에서 보여주는 "이 게임에서 얻은 점수"는 **기능 7의 `scoreLog`를 `gameId`로
  필터·집계**해서 계산한다. 점수 부여 기록을 단일 출처로 재사용한다.
- 게임 정의(이름·유형·배점·설명)는 `games` 컬렉션에서 읽는다(데이터 모델은 `04-data-model.md`).

### 3.6 타이머 제어

제한시간/스톱워치가 필요한 게임용. 게임 정의 `games.timer`로 기본 모드가 정해지고,
운영자가 진행 중 제어한다. 상태는 `state/current.timer`에 기록되어 모든 화면에 동기화.

| 동작 | 효과 |
| --- | --- |
| 시작 | `status='running'`, `startedAt=serverTimestamp` |
| 일시정지 | `status='paused'`, 경과분 `accumulatedSec`에 누적 |
| 재개 | `status='running'`, `startedAt` 재설정(누적 유지) |
| 리셋 | `status='idle'`, 누적·시작시각 초기화 |

- **카운트다운**: 남은 시간 0 도달 시 `status='finished'`(화면 점멸/알림). 윷놀이 30분·이어서그리기 20분 등.
- **스톱워치**: 정지까지 카운트업. 릴레이게임은 **측정 도구로만** 사용 — 운영자가 시간을 보고
  순위(200/100/50)를 점수 배정(기능 5)으로 입력(자동 기록 아님).
- 클라이언트는 `startedAt` 기준으로 시간을 로컬 계산해 표시(`04-data-model.md` §3).

### 3.7 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 첫 게임에서 "이전" | 비활성화(첫 게임 고정) |
| 마지막 게임에서 "다음" | 비활성화 또는 "전체 종료" 안내 |
| 자유 점프로 이미 진행한 게임 복귀 | 허용. 점수는 `scoreLog`에 누적돼 있어 중복 부여 주의(운영자 판단) |
| 게임 전환 시 타이머 | 새 게임의 `games.timer` 기본값으로 리셋(`idle`) |

### 3.8 진행 탭 UI — 목록 → 게임 진행 (가이드 단계)

운영 진행 탭은 두 뷰로 구성한다.

- **게임 목록(기본)**: 9게임을 순서대로 표시. 각 항목에 상태 뱃지(예정/진행중/완료)와
  버튼([게임 시작] / 현재 게임이면 [이어서 진행]).
  - 진행중 = `currentGameId`인 게임. 완료 = `state.finishedGameIds[]`에 포함.
- **게임 진행(진입 시)**: 상단 [← 목록] + 게임명 + 단계 인디케이터(`소개 → 진행 → 결과`).
  - **가이드형**: [다음 단계 →]/[← 이전 단계] 버튼으로 `phase`를 순차 전환.
  - 소개: 규칙 + 점수 산정 방식 + 타이머. 진행: 엔진/점수 컨트롤 + 타이머 + 되돌리기.
    결과: 이 게임 획득 점수 + **[게임 종료]** 버튼.
- **[게임 시작]** = `setCurrentGame`(currentGameId 설정 + phase=intro + 엔진 리셋) → 공개 화면 동기화.
- **[게임 종료]** = `finishGame`(`finishedGameIds`에 추가) → 목록으로, 완료 표시.
- 목록↔진행 전환은 운영자 화면 로컬 내비게이션(공개 화면은 항상 `state`를 따른다).

## 기능 4: 문제 제어 (퀴즈 진행 엔진)

### 4.1 목적 / 범위

퀴즈 게임(과자이름·초성·확대샷·노래·영화) 진행 중, 운영자 조작이 **진행 페이지에 같은
번호의 화면으로 동기화**되도록 한다. 단순 정답 공개 토글이 아니라, 문제마다
`q1 → q2 → q3` 화면 시퀀스를 운영자가 전진시키는 진행 엔진이다.

문제 내용 등록(CRUD)은 기능 6, 점수 부여의 데이터 기록은 기능 7(`scoreLog`)에서 정의한다.

### 4.2 상태 모델 (기능 3 `state/current` 확장)

```ts
state/current = {
  currentGameId, phase,                 // 기능 3
  currentQuestionId: string | null,     // 현재 문제
  quizScreen: 'q1' | 'q2' | 'q3',       // 현재 화면 번호 (역할별로 내용 다름)
}
```

- `answerRevealed` 불리언은 폐기하고 `quizScreen` 화면 포인터로 대체한다.
- 운영자가 `quizScreen`을 바꾸면 진행 페이지가 즉시 같은 번호 화면으로 전환된다.

### 4.3 화면 동기화 (q1/q2/q3)

화면 번호는 같고, **역할에 따라 내용이 다르다.**

| 화면 | 진행 페이지(참가자·빔) | 관리자 페이지 |
| --- | --- | --- |
| **q1** | 문제 표시 (텍스트/이미지) | 문제 + 정답 표시 + `정답자 등록` 버튼 |
| **q2** | **정답** 표시 | 사용자 목록 → 정답자 선택 + 점수 배정 → `등록` 버튼 |
| **q3** | "준비해주세요" 대기 화면 | "다음 문제로?" (→ q1 다음 문제 / → 팀별 맞춘 갯수) |

### 4.4 전체 퀴즈 흐름

```
퀴즈 소개(intro) → 연습 문제 → 본 문제[ q1 → q2 → q3 ]× N → 팀별 맞춘 갯수 공개(+점수 순위)
```

- "퀴즈 소개"는 게임 phase `intro`.
- "팀별 맞춘 갯수 공개(+점수 순위)"는 게임 phase `result`.
- 본 문제 구간은 phase `playing` + `quizScreen` 순환.

### 4.5 정답자 등록 (q2)

1. q1에서 `정답자 등록` → q2로 전환(진행 페이지엔 정답 공개).
2. 운영자가 **사용자 목록에서 정답자를 선택(여러 명 가능)** 하고 **점수를 배정**.
   - 점수 기본값은 문제의 `points`(기본 10), 등록 시 조정 가능.
   - 여러 명 선택 시 각자에게 점수 부여 → **각자의 팀 점수에 가산**.
3. `등록` 클릭 시:
   - 정답자 각각에 대해 `scoreLog`를 **결정적 docId `${gameId}__${questionId}__${userId}`로 set**
     (`gameId`, `teamId`, **`userId`**, **`questionId`**, `points`). → 더블클릭·중복 등록 멱등(기능 7.5).
   - 팀 점수는 `scoreLog` 합계로 파생(별도 누계 갱신 없음, 기능 7.2).
   - 개인의 "맞춘 문제"는 별도 테이블 없이 `scoreLog`(userId+questionId)로 표현.
4. 등록 후 q3(다음 문제 묻기)로 전환.

### 4.6 문제 타입: 연습 vs 본 문제

- 문제에 타입 필드: `kind: 'practice' | 'real'`.
- **연습 문제는 점수 미집계** (정답자 등록 단계 없이 q1→q2 정답 확인만, 또는 등록 버튼 비활성).
- 팀별 맞춘 갯수·점수 순위는 `kind:'real'` 문제만 집계.

### 4.7 이미지 파이프라인 (S3 + Lambda 서명서버)

확대샷·과자사진 등 이미지 문제용. 브라우저는 자격증명 없이 S3에 직접 업로드할 수 없어,
presigned URL을 발급하는 최소 백엔드(Lambda + API Gateway)를 둔다.

```
[등록] 관리자 업로드
  → 서명서버가 presigned PUT 발급
  → 브라우저가 S3로 직접 PUT 업로드
  → 서명서버가 presigned GET(유효기간 3일) 발급
  → 문제 문서에 promptImage/answerImage = { s3Key, url, expiresAt } 저장

[게임] 진행 페이지가 문제의 promptImage.url / answerImage.url 로 이미지 표시
```

> 필드명은 `{ s3Key, url, expiresAt }`로 통일(데이터모델 §7, 기능 6.2와 일치).

- 사진 등록은 **관리자 페이지(문제 관리, 기능 6)** 에서 수행.
- presigned GET URL은 3일 유효 — 행사 기간을 커버. 만료 시 서명서버로 재발급.
- 서명서버 상세(엔드포인트·IAM·버킷 정책)는 별도 인프라 문서에서 다룬다.

### 4.8 집계 (scoreLog 단일 출처 재사용)

- **개인 맞춘 문제**: `scoreLog`에서 `userId == 본인 && gameId == quiz` 카운트.
- **팀별 맞춘 갯수**: `scoreLog`의 quiz 항목을 `teamId`로 그룹·카운트.
- 별도 집계 컬렉션을 만들지 않는다(`scoreLog`에 `userId`·`questionId` 포함이 전제).

### 4.9 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| q2에서 정답자 0명 등록 | 허용("정답 없음"으로 등록) → q3로. 점수 변동 없음 |
| 같은 문제 중복 출제 | 문제 상태(§4.10)로 방지·경고 |
| 이미지 presigned URL 만료 | 서명서버 재발급 후 문서 갱신 |
| 등록 후 정답자 정정 | 기능 7 `되돌리기`로 해당 `scoreLog` 취소 후 재등록 |
| 연습 문제에서 점수 등록 시도 | 등록 버튼 비활성(집계 제외) |

### 4.10 문제 진행 규칙 및 상태

- **출제 단위**: 운영자가 카테고리를 선택하면 그 카테고리의 `questions`를 `order` 오름차순으로 진행.
- **다음 문제**: 현재 카테고리에서 `order` 다음이며 `used == false`인 문제로 이동, `quizScreen='q1'`.
  카테고리 소진 시 운영자가 다른 카테고리 선택 또는 phase를 `result`로 전환.
- **`used` 전환 시점**: 문제를 **q1로 띄우는 순간 `used = true`**(출제 시작).
- **상태 파생** (스키마는 `used:boolean` + `state.currentQuestionId` 둘로 표현):
  - 미출제 = `used == false`
  - 출제중 = `id == state.currentQuestionId`
  - 완료 = `used == true && id != currentQuestionId`
- 중복 출제는 `used == true` 문제를 목록에서 비활성/경고 처리해 방지.

## 기능 4-B: 제시어 게임 진행 엔진

### 4B.1 목적 / 범위

제시어형 게임(몸으로말해요, 이어서그리기 등)의 진행. 퀴즈 엔진(q1~q3)과 형제 격으로,
운영자 조작이 진행 페이지에 **같은 번호 화면(w1~w4)** 으로 동기화된다.
게임 시작 시 **제시어 묶음 3개를 3팀에 랜덤 배정**하고, 각 팀이 자신의 묶음 제시어를
순서대로 진행한다.

### 4B.2 데이터 모델 (제시어)

```ts
// 제시어 묶음 (게임당 3개)
promptSets/{id} = {
  gameId: string,        // 제시어 게임 id
  label?: string,        // 묶음 표시명(선택)
  order: number,
}

// 제시어 (묶음에 속함, 자유 개수)
prompts/{id} = {
  setId: string,
  text: string,          // 제시어
  category: string,      // 카테고리
  order: number,
}
```

### 4B.3 상태 모델 (state/current 확장)

```ts
state/current = {
  ...,
  promptScreen?: 'w1' | 'w2' | 'w3' | 'w4',
  promptAssignment?: { J: string, I: string, L: string },  // 팀→묶음 랜덤 배정 결과
  promptTeamOrder?: ('J' | 'I' | 'L')[],                    // 팀 진행 순서(게임 시작 시 고정)
  currentTeamId?: 'J' | 'I' | 'L',                          // 현재 진행 팀
  currentPromptId?: string,                                 // 현재 제시어
}
```

### 4B.4 게임 시작(초기화) 액션

운영자의 **"제시어 게임 시작"** 액션이 진행 상태를 초기화한다(phase가 `intro`→`playing` 전환 시점).

1. **묶음 배정 롤**: 해당 게임의 묶음 3개를 3팀에 랜덤 배정 → `promptAssignment`.
2. **팀 순서 고정**: `promptTeamOrder` 설정(예: `['J','I','L']`).
3. **첫 위치**: `currentTeamId = promptTeamOrder[0]`, `currentPromptId =` 그 팀 묶음의
   최소 `order` 제시어, `promptScreen = 'w1'`.
- **멱등성**: `promptAssignment`가 이미 있으면 재롤하지 않는다(자유 점프로 재진입해도 보존).
  운영자가 명시적 **"재배정"** 을 누를 때만 다시 롤.

### 4B.5 진행 로직 (팀별 일괄)

**한 팀이 자기 묶음 제시어를 모두 끝낸 뒤 다음 팀**으로 넘어간다(라운드로빈 아님).

- 현재 팀 묶음 = `promptSets/{promptAssignment[currentTeamId]}` → 소속 `prompts`를 `order` 오름차순.
- **다음 제시어**: 현재 팀 묶음에서 `currentPromptId` 다음 `order` 제시어로 이동, `promptScreen='w1'`.
- **팀 소진**(다음 order 없음): `promptTeamOrder`에서 다음 팀으로 → 그 팀 묶음 첫 제시어, `w1`.
- **모든 팀 소진**: phase를 `result`로 전환.
- "이미 출제" 여부는 `order` 순차 진행이라 별도 마킹 불필요(완료 = `order` < 현재).

### 4B.6 화면 동기화 (w1~w4)

| 화면 | 진행 페이지(참가자) | 관리자 |
| --- | --- | --- |
| **w1** | 제시어 순서(현재 팀 차례) 표시 | 팀/제시어/카테고리 표시 + `제시어 표시` |
| **w2** | 제시어 + 카테고리 공개 | (동일) + `제시어 가리기` |
| **w3** | 카테고리만 노출 | (동일) + `결과 입력` |
| **w4** | 제시어 + 카테고리 공개 | 정답 여부 선택 → 정답 시 즉시 점수 부여 + 맞춤 마킹 |

### 4B.7 w4 점수 부여 (항상 즉시)

- 정답이면 **즉시 점수 부여**: `scoreLog` append(`gameId`, `teamId`, **`promptId`**, `points`). 팀 점수는 합계로 파생(기능 7.2).
- 부여 점수는 게임 정의의 정답당 값(예: 이어서그리기 50) 또는 운영자 입력.
- "맞춤 마킹"은 `scoreLog`의 `promptId`로 표현(별도 테이블 없음).
- 등록(정답/오답) 후 §4B.5 진행 로직으로 다음 제시어/팀 이동(`w1`).
- 중복 등록 방지: `scoreLog` docId를 `${gameId}__${promptId}__${teamId}`로 고정(set) → 더블클릭 멱등(기능 7.5).

### 4B.8 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 묶음이 3개 미만 | 게임 시작 차단(경고: 묶음 3개 필요) |
| w4 오답 선택 | 점수 변동 없이 §4B.5로 다음 제시어 이동 |
| 잘못 부여 | 기능 7 `되돌리기`로 해당 `scoreLog` void |
| 팀이 묶음 제시어를 모두 소진 | `promptTeamOrder` 다음 팀으로, 모든 팀 종료 시 `result` 단계로 |
| 자유 점프로 재진입 | `promptAssignment` 보존(재롤 안 함), 마지막 위치 유지 |

## 기능 5: 점수 배정

### 5.1 목적

퀴즈 외 게임들의 점수를 부여한다. 운영 화면은 **현재 게임의 점수 유형**(`games.scoringType`)을
읽어 알맞은 컨트롤을 자동 렌더링한다. 모든 점수 부여는 기능 4와 동일하게
**`scoreLog` append** 단일 출처를 따르며, 팀 점수는 그 합계로 파생된다(기능 7.2).

### 5.2 배점 패러다임 (3유형 + 퀴즈형)

| 유형 | 방식 | 비고 |
| --- | --- | --- |
| `quiz` | 맞춘 사람의 팀에 점수 (기능 4) | 정답자 등록 |
| `rank` | 1·2·3등 팀 지정 → 순위별 사전 배점 자동 할당 | **공동 순위 허용** |
| `increment` | 팀 선택 → +N 프리셋 버튼 | 누적 |
| `free` | 운영자가 팀별 점수 직접 설정하여 할당 | 총점 옵셔널, 설정 시 합계 검증 |

### 5.3 순위형(rank)

- 운영자가 각 팀에 등수(1/2/3)를 지정 → 각 팀은 `rankPoints[지정등수]` 만큼 획득.
- **공동 순위 허용**: 같은 등수로 지정된 팀들은 동일 배점을 받는다(예: 공동 1등 두 팀 모두 1등 점수).
- 다음 등수 건너뜀 여부는 운영자 재량(시스템은 지정된 등수의 점수만 부여).
- **멱등 등록**: 라운드 단위 1회 성격 → `scoreLog` docId를 `${gameId}__${round||'main'}__${teamId}`로
  set(재지정 시 덮어쓰기). 더블클릭/재등록 안전(기능 7.5).

### 5.4 누적형(increment)

- 팀 선택 → `incrementOptions`의 프리셋 버튼(예: `[50]`, `[20]`)으로 가산.
- 같은 팀에 여러 번 가산 가능(이어서그리기 3회 등) → 매번 `scoreLog` 1건.
- **누적형은 의도적으로 멱등이 아니다**(반복 가산이 정상). 따라서 결정적 docId를 쓰지 않고
  **auto-id + 등록 버튼 디바운스/비활성**으로 더블클릭만 방지(기능 7.5).

### 5.5 가변형(free)

- 운영자가 **각 팀의 점수를 직접 입력하여 한 번에 배분**(음수 허용).
- **총점(`totalPoints`)은 옵셔널**:
  - 설정된 경우 → `등록` 시 **Σ(팀별 할당) == totalPoints** 검사, 불일치면 경고 후 차단.
  - 미설정(null)인 경우 → 합계 검증 없이 자유 분배(병뚜껑컬링처럼 총점 없는 게임).
- 미정 총점 게임은 운영자가 현장에서 총점을 먼저 설정할 수도 있다.
- **라이어게임**: 게임을 모두 진행한 뒤 운영자가 **팀별 최종 점수를 한 번에 배분**하는 `free` 게임.
  총배점이 정해져 있으면 합계 검증을 받는다. (점수를 두 팀 사이에서 +50/−50으로 *옮기는* 연결형
  트랜잭션이 아니라, **팀별 독립 배분 1회**이므로 되돌리기도 팀별 항목 단위로 안전.)
- **멱등 등록**: free 배분은 게임당 1회 성격 → `scoreLog` docId를 `${gameId}__free__${teamId}`로
  set(재배분 시 덮어쓰기). 더블클릭/재등록 안전(기능 7.5).

### 5.6 세부 라운드(sub-game)

- 게임 정의에 `rounds`(세부 종목 목록)가 있으면, 운영자가 라운드를 선택한 뒤 점수를 부여하고
  `scoreLog`에 라운드 라벨을 남긴다.
- **보드게임** = 순위형 × 3라운드(루미큐브·로보77·할리갈리), 라운드별 1등100/2등50/3등10.
- **릴레이게임** = 5종을 묶어 한 번만 순위(단일, 라운드 없음).

### 5.7 게임 정의 메타데이터

```ts
games/{id} = {
  name: string,
  description: string,
  scoringType: 'quiz' | 'rank' | 'increment' | 'free',
  totalPoints?: number,                          // 결과 표시·가변형 합계 검증 (옵셔널)
  rankPoints?: { 1: number, 2: number, 3: number },  // rank
  incrementOptions?: number[],                   // increment
  rounds?: string[],                             // 보드게임 등 세부 종목
  order: number,                                 // 진행 순서
}
```

### 5.8 scoreLog 기록 형식 (기능 7과 공유)

```ts
scoreLog/{id} = {
  gameId: string,
  teamId: 'J' | 'I' | 'L',
  points: number,            // 음수 가능(가변형 뚜기)
  userId?: string,           // 퀴즈 정답자 (개인 집계용)
  questionId?: string,       // 퀴즈 문제
  promptId?: string,         // 제시어 게임 정답 제시어
  round?: string,            // 세부 라운드 라벨
  rank?: 1 | 2 | 3,          // 순위형
  createdBy: 'admin',
  createdAt: Timestamp,
  voided: boolean,           // 되돌리기 시 true (기능 7)
}
```

### 5.9 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 가변형 합계 불일치(총점 설정 시) | 경고 후 차단, 운영자가 맞출 때까지 등록 불가 |
| 가변형 총점 미설정 | 합계 검증 없이 등록 허용 |
| 순위형 동점 지정 | 허용(공동 순위, 동일 배점) |
| 같은 라운드에 중복 부여 | 경고(이미 해당 라운드 기록 있음) |
| 잘못 부여 | 기능 7 `되돌리기`로 해당 `scoreLog` void 처리 |

## 기능 6: 콘텐츠 관리 (퀴즈 문제 + 제시어)

### 6.1 목적

운영자가 행사 전/중에 **퀴즈 문제**와 **제시어 묶음**을 등록·수정·삭제한다.
이미지(문제·정답)는 기능 4.7의 S3 + Lambda 서명 파이프라인으로 업로드한다.

### 6.2 퀴즈 문제 스키마

문제와 정답 각각 텍스트·이미지를 가질 수 있다(정답은 주로 이미지).

```ts
questions/{id} = {
  gameId: 'quiz',
  category: '과자이름' | '초성' | '확대샷' | '노래' | '영화',
  kind: 'practice' | 'real',                 // 연습/본문제
  promptText?: string,                       // 문제 텍스트(초성·제목·힌트 등)
  promptImage?: { s3Key, url, expiresAt },   // 문제 이미지(확대샷 등, 선택)
  answerText?: string,                       // 정답 텍스트
  answerImage?: { s3Key, url, expiresAt },   // 정답 이미지(주로 사용)
  points: number,                            // 기본 10
  used: boolean,                             // 출제 완료(출제중은 state에서 파생)
  order: number,
}
```

> 음원/영상(노래·영화)은 앱 내 재생하지 않는다. 문제는 텍스트/이미지로 표시하고
> 미디어는 운영자가 외부 스피커 등으로 재생한다.

### 6.3 제시어 스키마

기능 4-B와 동일.

```ts
promptSets/{id} = { gameId, label?, order }
prompts/{id}    = { setId, text, category, order }
```

### 6.4 제공 동작

| 대상 | 동작 |
| --- | --- |
| 퀴즈 문제 | 카테고리별 목록, 추가/수정/삭제, 순서 변경, 연습/본문제 지정, 이미지 업로드 |
| 제시어 묶음 | 게임별 묶음 3개 관리, 묶음 내 제시어(텍스트+카테고리) 자유 개수 추가/수정/삭제 |

### 6.5 이미지 업로드 (S3 파이프라인 재사용)

- 문제·정답 이미지 모두 동일 흐름: 업로드 → 서명서버 presigned PUT → S3 PUT → presigned GET(3일) → 문서에 `{s3Key, url, expiresAt}` 저장 (기능 4.7).
- URL 만료 시 서명서버로 재발급.

### 6.6 일괄/검증

| 항목 | 처리 |
| --- | --- |
| 텍스트 문제 일괄 등록 | `문제[Tab]정답` 형식 여러 줄 붙여넣기 지원(초성·노래·영화 등 다건). 이미지 문제는 개별 등록 |
| 필수값 | 퀴즈: category·kind·(promptText 또는 promptImage)·(answerText 또는 answerImage). 제시어: text·category |
| 묶음 개수 | 제시어 게임은 묶음 3개 권장(미만 시 게임 시작 차단, 기능 4-B) |

### 6.7 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 출제 완료(`used`) 문제 수정 | 허용하되 경고(이미 출제됨) |
| 이미지 없이 확대샷 저장 | 차단(확대샷은 이미지 필수) |
| 제시어 0개인 묶음 | 경고(게임 진행 시 빈 묶음) |
| 문제/제시어 삭제 | 확인 다이얼로그. 이미 점수가 부여된 경우 `scoreLog`는 유지(이력 보존) |

## 기능 7: 되돌리기

### 7.1 목적

잘못 부여한 점수를 취소한다. 모든 점수 변경은 `scoreLog` 단일 출처에 기록되므로,
되돌리기는 그 기록을 **무효화(void)** 하는 것으로 충분하다.

### 7.2 핵심 원칙: 점수는 파생값

- **팀 점수는 저장하지 않는다.** `scoreLog`에서 `voided == false`인 항목의 `points` 합으로
  실시간 계산한다. (팀별 그룹 합)
- 따라서 되돌리기 = 해당 항목 `voided = true` → 합계가 자동 재계산되어 즉시 반영.
- 모든 화면(리더보드·결과·집계)은 동일하게 "유효 `scoreLog` 합계"를 사용 → 불일치 없음.

### 7.3 제공 동작

| 동작 | 설명 |
| --- | --- |
| 최근 기록 목록 | `scoreLog` 최신순. 게임·팀·점수·시각·대상(정답자/제시어/라운드) 표시 |
| 취소(void) | 선택 항목 `voided=true`. 팀 점수·집계에서 즉시 제외 |
| 취소 복구(선택) | `voided=false`로 되돌림(실수로 취소한 경우) |
| 필터 | 게임별·팀별로 기록 필터링 |

### 7.4 집계에 미치는 영향

- 팀 총점: `Σ points (voided=false)` by team.
- 퀴즈 개인 맞춘 문제 / 팀별 맞춘 갯수: 동일하게 `voided=false`인 `userId`/`questionId` 기록만 집계.
- 게임별 결과(`result` 단계): `gameId`로 필터 + `voided=false` 합계.

### 7.5 중복 방지(멱등) 규약

동시성·더블클릭·두 기기 동시 등록으로 인한 **점수 중복 누적**을 막기 위해, 자연 키가 있는
부여는 **결정적 docId로 `set`**(중복 시 덮어쓰기), 반복이 정상인 부여만 `add`를 쓴다.

| 부여 유형 | docId 규약 | 방식 |
| --- | --- | --- |
| 퀴즈 정답자 | `${gameId}__${questionId}__${userId}` | set (멱등) |
| 제시어 정답 | `${gameId}__${promptId}__${teamId}` | set (멱등) |
| 순위(rank) | `${gameId}__${round||'main'}__${teamId}` | set (멱등) |
| 가변(free) | `${gameId}__free__${teamId}` | set (멱등, 재배분 덮어쓰기) |
| 누적(increment) | auto-id | add + 버튼 디바운스(반복 정상) |

- 공통적으로 등록 버튼은 처리 중 **비활성/디바운스**해 더블클릭을 1차 차단한다.
- docId에 들어가는 라운드 라벨은 `__` 등 구분자 충돌을 피하도록 **정규화**(슬러그화)해 사용한다.

### 7.6 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 이미 취소된 항목 재취소 | 무시(멱등) |
| 가변형 음수 항목 취소 | 합계에서 음수도 함께 제외(정상 재계산) |
| 대량 취소 | 항목별 void(일괄 선택 지원 가능) |
| 행사 종료 후 정정 | 동일하게 void로 처리(이력 보존) |

### 7.7 데이터 모델 영향 (앞 기능 정리)

- `teams` 문서에서 **`score` 필드 제거** — 점수는 `scoreLog` 합계로 파생.
  `teams`는 `id`·`name`·`color`(표시 메타)만 보유.
- 기능 4·4-B·5의 "팀 점수 가산"은 **`scoreLog` append**를 의미하며, 별도 누계 갱신은 없다.

## 기능 8: 행사 open/close 제어 (Remote Config)

### 8.1 목적

접근 제어를 Firestore 보안 규칙이 아닌 **앱 레벨 + Remote Config**로 한다. 운영자가
Firebase 콘솔에서 `app_status`를 토글하고, `closed`면 모든 페이지가 로고만 표시하며 로그인·조작이 막힌다.

### 8.2 동작

- **Firebase 콘솔 Remote Config**에서 `app_status` 토글(open ↔ closed). **앱 내 토글 UI 없음**.
- 모든 페이지는 RC를 로드 시 + 주기 폴링으로 페치 → `closed`면 로고 화면.
- **웹은 실시간 아님**: 콘솔 토글이 다음 페치 시점에 반영(open/close는 몇 번뿐이라 허용). `04-data-model.md` §2.
- `closed` 상태:
  - 로그인 페이지: 로고만, 이름 입력/로그인 불가.
  - 진행 페이지·운영 페이지: 로고 화면.

### 8.3 데이터

Remote Config 키: `app_status`('open'|'closed'), `logo_url`, `event_name` (상세 `04-data-model.md` §2).

### 8.4 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 폴링 지연으로 일부 폰이 늦게 반영 | 허용(open/close는 소수 이벤트). 빠른 반영 필요 시 새로고침 |
| 행사 완전 종료 | 콘솔에서 Firestore 쓰기 규칙을 false로 동결(최종 잠금) |

## 기능 9: 게임 관리(CRUD)

### 9.1 목적

게임을 코드에 고정하지 않고 **관리자가 직접 정의**한다. 게임 종류·진행 방식·점수 방식·배점·
산정 방식 문구·타이머를 모두 데이터로 관리한다(`games` 컬렉션, `04-data-model.md` §6).

### 9.2 두 축 분리

| 축 | 필드 | 값 |
| --- | --- | --- |
| 진행 방식 | `engineType` | `quiz`(q1~q3) / `prompt`(w1~w4) / `none`(일반) |
| 점수 방식 | `scoringType` | `quiz` / `rank` / `increment` / `free` |

- 관리자가 두 축을 자유 조합(예: `prompt`+`increment` = 이어서그리기).

### 9.3 제공 동작

| 동작 | 설명 |
| --- | --- |
| 목록/순서 | 게임 목록 표시, `order` 드래그/수정으로 진행 순서 변경 |
| 추가/수정 | 아래 폼(§9.4)으로 생성·편집 |
| 삭제 | 확인 다이얼로그. 연결된 문제/제시어/점수기록 처리 안내 |
| **프리셋 불러오기** | "기본 9게임 불러오기"로 예시 프리셋(`04-data-model.md` §6.1) 일괄 생성 후 편집 |

### 9.4 편집 폼 (유형 따라 적응)

- **공통**: 이름, 규칙 설명(`description`), **점수 산정 방식 문구(`scoringExplanation`)**,
  `engineType`, `scoringType`, `order`, 타이머(`timer`: mode + durationSec, 선택), `rounds`(선택).
- **scoringType별 추가 입력**:
  - `rank` → `rankPoints { 1, 2, 3 }`
  - `increment` → `incrementOptions[]`(정답/회당 점수, 예: `[50]`, `[20]`)
  - `free` → `totalPoints?`(옵셔널, 가변형 합계 검증용)
  - `quiz` → 별도 입력 없음(문제는 기능 6에서 관리)
- **engineType별 연결**:
  - `quiz` → 문제 관리(기능 6)와 연결
  - `prompt` → 제시어 묶음 관리(기능 6)와 연결, 묶음 3개 권장
  - `none` → 추가 연결 없음

### 9.5 검증

| 항목 | 규칙 |
| --- | --- |
| 필수값 | 이름·engineType·scoringType·order |
| rank | `rankPoints` 1·2·3 모두 입력 |
| increment | `incrementOptions` 최소 1개 |
| prompt 게임 | 진행 전 묶음 3개 필요(미만 시 경고, 기능 4-B) |

### 9.6 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 진행 중 게임 정의 수정 | 허용하되 경고(현재 진행에 영향) |
| 게임 삭제 시 점수 기록 존재 | `scoreLog`는 이력 보존, 결과/집계에서 해당 게임만 제외 |
| 프리셋 재불러오기(중복) | 기존 동일 gameId는 건너뛰고 요약 |
| engineType↔scoringType 부조합 | 허용(자유 조합). 단 quiz/prompt 엔진은 콘텐츠 연결 안내 |
