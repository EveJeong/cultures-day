# 인증 / 로그인 · 온보딩 설계

> 모든 사용자(참가자·운영자)의 공통 진입점. 이름 기반 로그인 + 최초 1회 온보딩(비번·닉네임 설정).
> 관련: `00-overview.md`, `02-admin-design.md`(사용자·팀 사전 등록), `04-data-model.md`(예정).

## 1. 목표

- 참가자는 **이름**으로 로그인하고, 최초 접속 시 본인의 **비밀번호·닉네임**을 직접 설정한다.
- 운영자(`admin`)는 별도 비밀번호로 운영 페이지에 진입한다.
- 참가자 명단과 팀 배정은 **운영자가 사전에 등록**해 둔 상태가 전제다. (등록 기능은 `02-admin-design.md`)

## 2. 화면 흐름

```
[1단계] 이름 입력 → "다음"
   │
   ↓ users/{이름} 조회  (이름 = 고유 docId)
   │
[2단계] 조회 결과에 따라 분기
   ├─ C. 이름 == "admin"            → 비밀번호 입력 → 일치 시 /admin 으로 이동
   ├─ A. passwordHash 있음(가입 완료) → 비밀번호 입력 → 일치 시 / (리더보드)
   ├─ B. passwordHash 없음(사전 배정) → 비밀번호 + 닉네임 설정 → 저장 후 / (리더보드)
   └─ D. 문서 없음(미등록)           → "등록되지 않은 이름입니다. 운영자에게 문의하세요"
```

- 1단계와 2단계는 같은 페이지에서 단계 전환(스텝 UI). 라우트는 `/login` 또는 진입 시 미인증이면 자동 노출.
- 2단계의 입력 폼은 분기(A/B/C)에 따라 다르게 렌더링한다.
  - A·C: 비밀번호 1칸
  - B: 비밀번호 + 비밀번호 확인 + 닉네임

## 3. 데이터 모델 (`users` 컬렉션)

```ts
// 문서 ID = name (이름이 고유키)
users/{이름} = {
  name: string,                 // 로그인 키. 운영자가 사전 입력
  teamId: 'J' | 'I' | 'L',      // 팀. 운영자가 사전 배정
  nickname: string | null,      // B 단계에서 본인이 설정. 가입 전엔 null
  passwordHash: string | null,  // SHA-256(평문). null이면 아직 미가입 → B 분기
  isAdmin: boolean,             // true면 C 분기 (운영자 계정)
}
```

- **A/B 분기 기준**: `passwordHash === null` 이면 B(최초 가입), 값이 있으면 A(가입 완료).
- **C 분기 기준**: `isAdmin === true`. (관례상 docId도 `"admin"`)
- **D 분기 기준**: 문서 자체가 존재하지 않음.

## 4. 확정된 결정

| 항목 | 결정 | 비고 |
| --- | --- | --- |
| 비밀번호 저장 | **클라이언트 SHA-256 해시 후 저장** | 원본 비번은 DB에 남기지 않음. `crypto.subtle.digest` 사용 |
| 미등록 이름 | **거부** | 운영자 문의 안내 메시지. 자가 가입 불가 |
| 동명이인 | **없음 가정** | 이름이 곧 docId (고유키) |

## 5. 세부 동작

### 5.1 admin 계정
- 운영자가 **사전 시드**로 생성: `users/admin = { isAdmin: true, passwordHash: <사전설정>, teamId: 임의/무관 }`.
- admin은 B(자가 가입) 분기를 타지 않는다. 비번은 시드 시점에 정해 둔다.

### 5.2 비밀번호 재설정
- 참가자가 비번을 잊으면, **운영자가 해당 유저의 `passwordHash`를 `null`로** 만든다.
- 그러면 그 사용자는 다음 로그인 때 다시 B(재설정) 분기를 탄다.
- 이 초기화 동작의 UI는 `02-admin-design.md`의 "사용자·팀 관리"에서 구현.

### 5.3 세션 유지
- 로그인 성공 시 `localStorage`에 `{ userId(=이름), role: 'participant'|'admin' }` 저장.
- 새로고침 시 세션이 있으면 로그인 단계를 건너뛰고 해당 페이지로 직행.
- **로그아웃** 버튼 제공 → 세션 삭제 후 1단계로.

### 5.4 닉네임 용도
- 리더보드에서 **본인 식별·표시용** (점수 집계는 팀 단위이나, "나 = 어느 팀"을 보여주는 데 사용).

## 6. 검증 규칙 (입력값)

| 필드 | 규칙 |
| --- | --- |
| 이름 | 공백 트림, 빈 값 불가 |
| 비밀번호 | 최소 길이(예: 4자 이상), B 단계는 "비밀번호 확인"과 일치해야 함 |
| 닉네임 | 공백 트림, 빈 값 불가 (중복 허용 — 식별키가 아니므로) |

## 7. 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 비번 불일치(A·C) | "비밀번호가 일치하지 않습니다" 표시, 재입력 |
| B 단계 중 새로고침 | 아직 `passwordHash` 미저장 상태이므로 다시 B 분기로 진입 (멱등) |
| 같은 사람이 두 기기에서 로그인 | 허용 (읽기 중심이라 충돌 없음) |
| 네트워크 일시 끊김 | Firestore 오프라인 캐시로 조회 시도, 실패 시 재시도 안내 |

## 8. 보안 한계 (명시)

- 클라이언트 해시 + 클라이언트 분기라, 기술적으로 우회 가능한 수준이다. **사내 1일 이벤트용 "장난 방지"** 목적임을 전제로 한다.
- Firestore 보안 규칙은 "읽기 공개 + 쓰기 허용(이벤트 기간)"으로 운영하며, 행사 종료 후 쓰기를 잠근다.
