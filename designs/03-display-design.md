# 진행(점수판) 페이지 상세 설계

> 빔프로젝터 + 참가자 폰이 함께 보는 읽기 전용 페이지. **개인화된 동일 뷰**(로그인 사용자의
> "내 팀"을 강조)이며, 빔프로젝터는 로그인 없는 관전 모드(`/spectator`)로 중립 표시한다.
> 관련: `04-data-model.md`(읽는 데이터), `02-admin-design.md`(상태를 만드는 운영), `01-auth-design.md`.

## 기능 목록 (작성 진행)

| # | 기능 | 상태 |
| --- | --- | --- |
| 1 | 상태 게이팅 / 라우팅 | ✅ 작성 |
| 2 | 리더보드(메인) | ✅ 작성 |
| 3 | 게임 진행 뷰(phase) | ✅ 작성 |
| 4 | 퀴즈 참가자 화면(q1~q3) | ✅ 작성 |
| 5 | 제시어 참가자 화면(w1~w4) | ✅ 작성 |
| 6 | 우리 팀 점수 산정 방식 | ✅ 작성 |
| 7 | QR 참가 유도 | ✅ 작성 |

## 공통 전제

- 모든 데이터는 Firestore 구독(`onSnapshot`)으로 실시간 반영. 진행 페이지는 **읽기 전용**.
- 구독 대상(Firestore): `state/current`, `teams`, `games`, `scoreLog`, 그리고 현재 화면에
  필요한 `questions`/`prompts`.
- 행사 상태(`app_status`)는 **Remote Config**에서 로드 시 + 주기 폴링으로 페치(`04-data-model.md` §2).
- 개인화 기준: 로그인 세션의 `userId` → `users/{userId}.teamId`로 "내 팀" 결정.

## 기능 1: 상태 게이팅 / 라우팅

### 1.1 목적

Remote Config `app_status`와 인증 상태에 따라 어떤 화면을 보여줄지 결정한다. 접근 제어는
Firestore 규칙이 아니라 이 앱 레벨 게이팅으로 수행한다(`04-data-model.md` §2, §12).

### 1.2 게이팅 우선순위

```
1) app_status == 'closed'  → (모든 라우트) 로고 화면. 로그인·조작 불가
2) 라우트별 분기:
   /          → 미인증: 로그인 화면 / 인증 참가자: 메인(개인화 뷰) / admin: /admin 안내
   /admin     → admin 계정만(아니면 로그인 또는 거부)
   /spectator → 로그인 없이 관전(중립) 뷰
```

- `closed` 체크가 **최상위**. 닫히면 어떤 경로든 로고만.

### 1.3 라우트

| 라우트 | 용도 | 인증 | 개인화 |
| --- | --- | --- | --- |
| `/` | 참가자 메인(진행 페이지) | 필요 | O (내 팀 강조) |
| `/spectator` | 빔프로젝터 관전 | 불필요 | X (중립) |
| `/admin` | 운영 | admin | — |

### 1.4 관전(spectator) 모드

- 빔프로젝터는 `/spectator`만 띄워두면 됨(로그인 번거로움 제거).
- **개인화 없음**: "내 팀" 강조·우리팀 점수 산정 방식 미표시. 순수 리더보드 + 현재 게임 진행 화면.
- 진행 상태(`state/current`)·점수는 참가자 뷰와 동일하게 실시간 반영.
- `closed`면 동일하게 로고 화면.

### 1.5 로고(closed) 화면

- **보노보노 + "바딧불이 야호"** 로고를 무지개 배경 위 중앙 표시(Remote Config `logo_url`·`event_name`).
- 로그인 입력·게임 화면 일절 미노출.
- `app_status`가 `open`으로 바뀌면 다음 RC 페치 시점에 정상 화면으로 전환.

### 1.6 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 진행 중 `closed`로 전환 | 다음 RC 페치 시점에 로고 화면(웹은 실시간 아님) |
| 세션은 있는데 명단에서 삭제됨 | "등록되지 않은 사용자" 처리 → 로그인 화면 |
| 세션 docId가 더 이상 해석 안 됨(이름 변경 등) | 삭제와 동일 처리 → 재로그인(`02-admin-design.md` §2.4) |
| `/`로 들어온 미인증 참가자 | 로그인 화면(`01-auth-design.md` 흐름) |
| 관전 URL을 참가자가 열람 | 허용(중립 뷰). 개인화는 `/`에서만 |

## 기능 2: 리더보드(메인)

### 2.1 목적

3팀의 점수 현황을 실시간 순위로 보여준다. 로그인 사용자는 "내 팀"이 강조된 개인화 뷰를,
빔프로젝터(관전)는 중립 뷰를 본다.

### 2.2 레이아웃 (점수바 상시 + 중앙 게임화면)

```
┌──────────────────────────────────────────┐
│  [점수 랭킹 바] J 320 · I 280 · L 250      │ ← 상시 노출(컴팩트)
├──────────────────────────────────────────┤
│                                          │
│        중앙: 현재 게임 진행 뷰            │ ← 기능 3~5(phase·q·w)
│        (result·대기 시 리더보드 전체 확대) │
│                                          │
└──────────────────────────────────────────┘
```

- 상단 **점수 랭킹 바**는 항상 노출. 중앙은 현재 게임 화면.
- `result` 단계나 게임 사이 대기에는 **리더보드가 전체 화면으로 확대**.
- 반응형: 빔프로젝터(가로 와이드), 폰(세로 스택).

### 2.3 점수 산정 (파생)

- 팀 총점 = `Σ scoreLog.points where teamId=T and voided=false` (`04-data-model.md` §9).
- `scoreLog` 구독 → 클라이언트에서 팀별 합산 → 내림차순 정렬.

### 2.4 표시 항목

| 항목 | 내용 |
| --- | --- |
| 순위 | 1·2·3위 (동점 시 공동 순위 표기) |
| 팀 | `teams.name` + `teams.color` |
| 점수 | 총점. 변동 시 카운트업/하이라이트 애니메이션 |

### 2.5 개인화 (내 팀 강조)

- **순위는 실제 그대로 유지**, 내 팀 행만 **테두리·배지·배경색**으로 강조.
- 헤더에 "나: {nickname} ({내 팀})" 표시.
- 관전(`/spectator`) 모드는 강조·배지 없이 중립.

### 2.6 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 모든 팀 0점(시작 전) | 0점 동률, 정의된 팀 순서(J·I·L)로 표시 |
| 동점 | 공동 순위(같은 등수 표기) |
| 점수 음수(가변형 뚜기 결과) | 그대로 표시(합계 반영) |
| 관전 모드 | 개인화 요소 전부 숨김 |

## 기능 3: 게임 진행 뷰 (phase 디스패처)

### 3.1 목적

중앙 영역이 `state.phase`와 현재 게임의 **`engineType`**(진행 방식)에 따라 무엇을 그릴지 결정한다.
점수 방식(`scoringType`)은 화면 선택에 관여하지 않는다(두 축 분리, `04-data-model.md` §6).
퀴즈/제시어 게임은 전용 화면(기능 4·5)으로, 그 외는 일반 진행 화면으로 위임한다.

### 3.2 디스패치 (phase × engineType)

| phase | 화면 |
| --- | --- |
| `intro` | 게임명 + 규칙(`games.description`) + **우리팀 점수 산정 방식**(기능 6) 크게 |
| `playing` | `engineType==='quiz'` → 퀴즈 화면(기능 4, q1~q3)<br>`engineType==='prompt'` → 제시어 화면(기능 5, w1~w4)<br>`engineType==='none'` → 일반 진행 화면(§3.4) |
| `result` | 이 게임 획득 점수(팀별) + 순위. 퀴즈면 팀별 맞춘 갯수 함께(§3.5) |

### 3.3 게임유형 판별 (engineType 기준)

- `engineType === 'quiz'` → 퀴즈 화면(기능 4, q1~q3).
- `engineType === 'prompt'` → 제시어 화면(기능 5, w1~w4).
- `engineType === 'none'` → 일반 진행 화면(§3.4).

> 진행 방식은 `engineType`, 점수 방식은 `scoringType`으로 분리(`04-data-model.md` §6).

### 3.4 일반 진행 화면 (엔진 없는 게임)

- 게임명 + 규칙 요약 + "진행 중" 표시.
- `rounds`가 있으면 현재 라운드 표기(보드게임 루미큐브 등).
- **타이머**(있으면) 표시(§3.6).
- 상단 점수바 유지. 실제 게임은 오프라인 진행, 화면은 안내·점수·타이머 역할.

### 3.5 result 화면

- `state.currentGameId`로 `scoreLog` 필터(`voided=false`) → 팀별 합산 → 이 게임 획득 점수·순위 발표.
- 퀴즈 게임은 팀별 맞춘 갯수(정답 기록 수)도 함께.
- 발표 후 전체 리더보드(기능 2)로 자연스럽게 연결.

### 3.6 타이머 표시

- `state.timer.mode`가 있으면 진행 뷰에 `MM:SS` 표시.
- **카운트다운**: 남은 시간. 막바지(예: 1분 이하) 빨강, `finished` 시 점멸/강조.
- **스톱워치**: 경과 시간 카운트업.
- 표시값은 `startedAt`(서버 시각) 기준 로컬 계산. **빔프로젝터 화면이 권위**이고 참가자 폰은
  기기 시계 오차만큼 근사치(`04-data-model.md` §3). `finished` 판정도 빔 화면 기준.
- `status==='idle'`이면 타이머 숨김 또는 초기값 표시.

### 3.7 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| `playing`인데 현재 문제/제시어 미설정 | "준비 중" 대기 표시 |
| result인데 이 게임 점수 0건 | "획득 점수 없음" 또는 0점 표기 |
| 타이머 `finished` | 화면 강조(시간 종료 안내) |
| `engineType` 미설정/미상 | 일반 진행 화면(`none`)으로 폴백 |

## 기능 4: 퀴즈 참가자 화면 (q1~q3)

### 4.1 목적

퀴즈 진행 엔진(운영 기능 4)의 **참가자/빔 측 렌더링**. `state.quizScreen`에 따라 화면을 전환한다.
운영 4.3의 참가자 컬럼을 구체화한 것.

### 4.2 데이터

- `state.currentQuestionId` → `questions/{id}` 로드.
- `state.quizScreen`('q1'|'q2'|'q3').
- 이미지: `promptImage.url` / `answerImage.url`(presigned, S3).

### 4.3 화면별 렌더링

| 화면 | 표시 |
| --- | --- |
| **q1** | 카테고리 라벨 + 문제(`promptText`/`promptImage`) 크게. **정답 숨김** |
| **q2** | 문제 유지 + **정답 공개**(`answerText`/`answerImage`, 주로 이미지) |
| **q3** | "준비해주세요" 대기 화면(다음 문제 준비) |

### 4.4 표시 요소

- **연습 문제**(`kind==='practice'`)는 "연습" 뱃지 표시(점수 미집계 안내).
- 카테고리(과자이름·초성·확대샷·노래·영화) 라벨 상시.
- 노래·영화: 미디어는 외부 재생, 화면은 "노래 맞추기 N번" + 공개 시 정답.
- 상단 점수바(기능 2) 유지.

### 4.5 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 이미지 URL 만료/로드 실패 | 대체 텍스트·플레이스홀더 표시 |
| `currentQuestionId` null | "준비 중" 표시 |
| q2인데 정답 데이터 없음 | 텍스트만/플레이스홀더 |

## 기능 5: 제시어 참가자 화면 (w1~w4)

### 5.1 목적

제시어 진행 엔진(운영 기능 4-B)의 **참가자/빔 측 렌더링**. `state.promptScreen`에 따라 전환.
운영 4B.6의 참가자 컬럼을 구체화한 것.

### 5.2 데이터

- `state.currentTeamId`, `state.currentPromptId` → `prompts/{id}` 로드.
- `state.promptAssignment`(팀→묶음) → 현재 팀 차례 표시.
- `state.promptScreen`('w1'|'w2'|'w3'|'w4').

### 5.3 화면별 렌더링

| 화면 | 표시 |
| --- | --- |
| **w1** | 제시어 순서/현재 팀 차례 표시(제시어·카테고리는 아직 숨김) |
| **w2** | 제시어 + 카테고리 **공개** |
| **w3** | **카테고리만** 노출(제시어 가림) |
| **w4** | 제시어 + 카테고리 공개(정답 확인) |

> 화면 번호는 운영자와 동일. w2·w4는 공개, w3는 카테고리만 — 출제자/팀 시야 분리를 위한 단계.

### 5.4 표시 요소

- 현재 진행 팀 강조(팀 색).
- 진행 페이지는 읽기 전용 — 정답 여부 입력은 운영자(w4)가 수행하며, 점수 반영은 점수바로 나타남.
- 상단 점수바 유지. 타이머가 있는 게임(이어서그리기 등)은 타이머 표시(기능 3.6).

### 5.5 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| `currentPromptId` null | "준비 중"/팀 대기 표시 |
| 팀 묶음 소진 | 다음 팀/결과 단계 안내 |
| 배정(`promptAssignment`) 없음 | "배정 대기" 표시 |

## 기능 6: 우리 팀 점수 산정 방식

### 6.1 목적

현재 게임에서 **우리 팀이 어떻게 점수를 얻는지**를 참가자에게 보여준다.
원 요구사항의 "우리팀의 점수 산정 방식" 항목.

### 6.2 데이터 / 표시

- 출처: 현재 게임의 `games.scoringExplanation`(점수 산정 설명 문구).
- **intro 단계**: 규칙과 함께 크게 표시(룰 숙지).
- **playing 단계**: 작게/접이식으로 상시 접근(참가자가 점수 획득법 재확인).

### 6.3 개인화

- `/`(로그인): "우리 팀은 …" 톤으로 표시(내 팀 색 강조).
- `/spectator`(관전): "각 팀은 …" 중립 톤.

### 6.4 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| `scoringExplanation` 비어있음 | `description`/배점 메타로 자동 요약 표시 |
| 가변형(미정 배점) | "운영자 배점" 등 안내 문구 |

## 기능 7: QR 참가 유도

### 7.1 목적

빔프로젝터 화면의 QR을 참가자가 스캔해 폰으로 접속(로그인 페이지)하도록 유도한다.

### 7.2 동작

- QR 인코딩 대상: 앱 베이스 URL(로그인 `/`).
- 생성: 클라이언트에서 QR 라이브러리로 렌더(서버 불필요).
- 노출 위치: 주로 `/spectator`(빔프로젝터). 짧은 URL 텍스트 병기.
- 게임 진행 중에는 작게(코너), 대기/시작 전에는 크게 표시할 수 있음.

### 7.3 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| `closed`(로고 화면) | QR 숨김 또는 "잠시 후 시작" 안내와 함께 표시(선택) |
| URL 환경별 상이(로컬/배포) | 배포 URL을 빌드 환경변수로 주입 |
